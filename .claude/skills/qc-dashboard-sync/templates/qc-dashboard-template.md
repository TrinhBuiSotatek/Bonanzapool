# QC Dashboard

> **Source of truth** cho trạng thái artifact của tất cả features/use cases trong dự án.
>
> **Ownership:**
> - `qc-dashboard-sync` là **SOLE WRITER** của mọi cột trong file này (kể cả các cột stt). Tạo file từ template khi chưa có; refresh dữ liệu khi user chạy `/qc-dashboard-sync` (top-down) hoặc `/qc-dashboard-sync <UC-ID>` (bottom-up).
> - Cột `Site`, `{{ID_LABEL}}`, `Folder ID`, `Module`, `Feature/Use case name`, `In scope?` lấy từ `site-map-handoff.md` (top-down) hoặc `Need confirm` (bottom-up).
> - Cột `Files stt` lấy từ disk scan 6 loại artifact.
> - Cột `UC review stt`, `Scenario design stt`, `TC design stt`, `UI extract stt` (nếu có) → `qc-dashboard-sync` tính theo 2 bước:
>   - **In-progress check**: nếu `.claude/skills/<skill>/process-logging/<UC-ID>/progress.md` tồn tại (skill đang chạy dở cho UC đó) → cell = giá trị `status:` đọc verbatim từ file.
>   - **Done-state derivation** (khi file in-progress không tồn tại): với `UC review stt` → parse `Tổng điểm` row trong audited file (CHỈ khi version mới hơn cached cell); với `Scenario design stt` / `TC design stt` → derive từ Files stt (không parse content); với `UI extract stt` → parse file ui-elements highest version (CHỈ khi mới hơn cached cell).
> - Cột `Execute stt` (placeholder cho `qc-execute` skill tương lai) — `qc-dashboard-sync` chỉ preserve verbatim từ user edit; chưa có skill nào ghi.
>
> **Các skill khác KHÔNG mở file này.** `qc-uc-read`, `qc-func-scenario-design`, `qc-func-tc-design`, `qc-ui-extract` chỉ duy trì checkpoint per-UC subfolder cho resume — tránh conflict khi nhiều skill chạy cùng lúc.
>
> **Sort order:** Dòng `Sorting: <v1> >> <v2> >> ...` ngay trên header. `qc-dashboard-sync` auto-generate khi level cao nhất (Site → Module) có >1 distinct value. User có thể edit thứ tự; skill sẽ honor edit. Levels thấp hơn (Module nếu sort theo Site, UC ID luôn) default ascending.
>
> **DO NOT delete rows.** Feature/UC ra khỏi scope vẫn giữ row, user có thể edit `In scope?` về `No` thủ công nếu cần.

## Ghi chú trạng thái

- **Site:** Portal/site mà feature/UC thuộc về (vd: Admin, Vendor, User, App, Web...).
- **{{ID_LABEL}}:** Định danh canonical của feature/UC theo site-map. Tên cột adapt theo cách dự án gọi (`Use Case ID`, `Feature ID`, `Story ID`, ...). Sau khi `qc-site-map` Mode 3 reconcile xong, đây là ID chính thức.
- **Folder ID:** ID/tên folder thực tế trên disk (do `qc-dashboard-sync` extract từ tên sub-folder khi scan). Dùng để map disk scan ↔ row.
  - Với row đi qua top-down chuẩn (folder name khớp canonical ID): `Folder ID` = `{{ID_LABEL}}` (self-reference).
  - Với row alias (folder name khác canonical, đã được Mode 3 reconcile): `Folder ID` giữ tên folder gốc, `{{ID_LABEL}}` được update về canonical từ site-map.
  - Với row bottom-up chưa reconcile: `Folder ID` = `{{ID_LABEL}}` = ID extract từ folder (chờ Mode 3 reconcile).
- **Module:** Module / nhóm chức năng.
- **Feature/Use case name:** Tên human-readable (do site-map cung cấp; bottom-up để trống chờ user / Mode 3 cập nhật).
- **In scope?:** `Yes` (in scope) / `No` (out of scope) / `Need confirm` (chưa reconcile bởi site-map Mode 3 — sẽ tự cập nhật sau khi Mode 3 chạy). `qc-dashboard-sync` KHÔNG tự đổi giá trị này; user có thể edit thủ công bất cứ lúc nào.
- **Files stt:** Chỉ liệt kê các loại file artifact đã TÌM THẤY trên disk cho UC. Loại file Missing KHÔNG được ghi vào cell. Format trong cell: mỗi loại tìm thấy là 1 dòng `<Type>: V<N>`, các dòng nối bằng `<br>`, theo thứ tự cố định (skip loại nào không có):
  ```
  Specs: V<N>
  WF: V<N>
  Audited: V<N>
  Scenario: V<N>
  TC md: V<N>
  TC xlsx: V<N>
  ```
  Ví dụ rendered:
  - Có một số file: `Specs: V2<br>WF: V1<br>Scenario: V1<br>TC md: V2`
  - Đã scan nhưng không tìm thấy file nào: literal `No files yet`
  - Chưa từng scan (row mới chưa được sync): cell để trống.

  Mỗi item tham chiếu folder qua path-registry logical name, lookup folder bằng giá trị `Folder ID` của row:
  - `Specs` ← `requirement-files/<Folder ID>/` (file `.md/.docx/.pdf`, không kể image, không kể `_extracted_`)
  - `WF` ← `requirement-files/<Folder ID>/` (file image: `.png/.jpg/.fig/.svg/...`)
  - `Audited` ← `uc-review-report/<Folder ID>/` (file `*_audited_*.md`)
  - `Scenario` ← `func-test-scenarios/<Folder ID>/` (file có `_scenarios_` trong tên)
  - `TC md` ← `func-test-cases-draft/<Folder ID>/` (file `_testcases_*.md`)
  - `TC xlsx` ← `func-test-cases/<Folder ID>/` (file `_testcases_*_v<N>.xlsx`)

- **Các cột stt (UC review / Scenario design / TC design / UI extract):** lấy từ `<skill>/process-logging/progress.md` qua best-effort parse (xem section "Process-log parse contract" trong SKILL.md). Format hiển thị:
  - `UC review stt` — `<Verdict> v<N> (Score X/100)` khi done (vd: `Ready v2 (Score 91/100)`, `Conditionally Ready v2 (Score 73.1/100)`, `Not Ready v1 (Score 45/100)`); raw `status` khi đang chạy (vd: `Running — Phase 3`, `Phase 3 done`); BLANK nếu chưa có entry.
  - `Scenario design stt` — `v<N> generated` khi done; raw `status` khi đang chạy; BLANK nếu chưa có entry.
  - `TC design stt` — `v<N> generated` (workflow generate-test-cases) hoặc `v<N> updated` (workflow update-test-cases) khi done; raw `status` khi đang chạy; BLANK nếu chưa có entry.
  - `UI extract stt` (cột optional, auto-inject khi `qc-ui-extract/progress.md` có entry đầu tiên) — `v<N> (audited v<M>)`; nếu multi-page → `v<min> (audited v<M>) [<K>p]`. BLANK nếu chưa có entry.
