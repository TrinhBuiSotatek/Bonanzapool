# US-EXBOT-006: Execute delta-only hedge adjustment

## Story Statement

As the **ExBot System Operator (hedge-sync worker)**,
I want to **adjust the HL short position by only the delta between target and actual size — never doing a full close/open cycle**,
So that **hedge cost and HL rate limit consumption are minimized for each adjustment**.

**INVEST check:** I✓ N✓ V✓ E✓ S✓ T✓

## Acceptance Criteria

```gherkin
AC-EXBOT-006-1: Happy path — delta-only adjustment executed and reconciled
  Given a hedge-sync message with reasons=['drift_threshold'] and stateVersion matching D1
  When the hedge-sync worker acquires the UserLockDO lease for this user
    And fetches actual HL position (clearinghouseState)
    And computes delta = targetShortEth - actualShortEth
  Then the worker submits only the delta adjustment (increase or decrease) via adjustShortDelta
    And waits for fill confirmation
    And runs post-order reconcile: fetches actual position, verifies size
    And updates bot_runtime_state.last_known_hl_short_size to reconciled value
    And re-computes stop_trigger_px and places/replaces stop via INV-STOP protocol
    And records a rebalance_attempts row with status='success'

AC-EXBOT-006-2: Edge case — stateVersion mismatch (concurrent update)
  Given the hedge-sync message carries stateVersion=5
    But bot_runtime_state.state_version in D1 is now 6 (updated by another worker)
  When the hedge-sync worker checks stateVersion before acquiring lock
  Then the worker discards the message and records attempt status='skipped'
    And does NOT submit any HL order

AC-EXBOT-006-3: Edge case — UserLockDO lease not acquired (another worker holds it)
  Given another hedge-sync worker already holds the UserLockDO lease for this user
  When the worker attempts to acquire the lease
  Then the attempt returns acquired=false with currentHolderExpiresAt
    And the worker re-queues the message with a delay (not an immediate retry)
    And does NOT submit any HL order

AC-EXBOT-006-4: Negative path — HL order rejected, circuit breaker incremented
  Given the worker submits the delta adjustment
  When HL returns an order rejection (e.g., insufficient margin)
  Then the worker records rebalance_attempts with status='failed' and error details
    And calls incrementCircuitBreaker for this hedge_leg_id
    And after 3 consecutive failures within 24h, circuit_breakers.state transitions to 'open'
    And enqueues a notification message to alert the investor
```

## Notes

Full close/open is FORBIDDEN in normal hedge-sync. Allowed only for target=0, emergency reset, or manual reset (Phase B+). Cloid is deterministic: `keccak256(bnza:{botId}:{attemptId}:{stage}:{version})`.
