# US-EXBOT-007: Rebalance LP range when price moves out of range

## Story Statement

As the **ExBot System Operator (hedge-sync worker handling range_out)**,
I want to **rebalance the Uniswap V3 LP position to a new ±5% range around the current price when the existing range is exited**,
So that **the LP continues earning fees and the hedge calculation remains accurate**.

**INVEST check:** I✓ N✓ V✓ E✓ S✓ T✓

## Acceptance Criteria

```gherkin
AC-EXBOT-007-1: Happy path — LP rebalance completes, bot returns to active
  Given a bot with lifecycle_state='active' and rangeState='above' (price above upper tick)
  When the light-check detects range_out and hedge-sync triggers LP rebalance
  Then the system sets lifecycle_state='lp_rebalancing'
    And calls BnzaExVault.vaultRebalance(tokenId, newTickLower, newTickUpper, slippageBps) via Operator
    And BnzaExVault internally executes: decreaseLiquidity 100% → collect fees → swap → NFPM.mint new range
    And the system receives the new tokenId from the VaultRebalanced on-chain event
    And D1 positions row is updated with new tokenId, tickLower, tickUpper, liquidity
    And the hedge is re-synced to the new LP ETH amount (delta-only)
    And lifecycle_state returns to 'active'

AC-EXBOT-007-2: Edge case — lp_operations recovery on worker crash
  Given a BnzaExVault.vaultRebalance tx was submitted (status='submitted' in lp_operations)
    But the worker crashed before reflecting the new tokenId in D1
  When the worker restarts and detects lp_operations with status='submitted'
  Then the worker fetches the tx receipt and decodes the VaultRebalanced event
    And recovers the new tokenId from the event
    And updates D1 positions + sets lp_operations.status='recovered'
    And allows normal monitoring to resume (lifecycle_state exits lp_rebalancing)

AC-EXBOT-007-3: Negative path — LP rebalance fails (slippage exceeded)
  Given a BnzaExVault.vaultRebalance call reverts due to slippage > tolerance
  When the on-chain tx fails
  Then the system does NOT assume any new LP state
    And sets lifecycle_state to 'safe_mode'
    And reconciles the hedge against the confirmed existing LP state
    And enqueues an admin alert notification

AC-EXBOT-007-4: Edge case — hedge adjustments blocked during rebalance
  Given lifecycle_state='lp_rebalancing'
  When a light-check or hedge-sync message arrives for this bot
  Then the worker skips all hedge mutations for this bot
    And does not attempt any HL order submission
```

## Notes

LP rebalance only proceeds when NOT in safe_mode/error/closing. The lp_operations idempotency ledger prevents double submission on worker restart. wethIndex per chain (Base vs Optimism) must be read from D1 positions — not assumed.
