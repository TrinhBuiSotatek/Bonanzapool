# US-EXBOT-009: Execute bot_safe_close and automatic re-entry

## Story Statement

As the **ExBot System Operator (close and recovery workers)**,
I want to **close an unsafe bot (hedge-first) and receive my funds back through the on-chain redemption queue**,
So that **my capital is returned safely even when the system must force-close my position**.

**INVEST check:** I✓ N✓ V✓ E✓ S✓ T✓

## Acceptance Criteria

```gherkin
AC-EXBOT-009-1: Happy path — bot_safe_close completes, funds returned via queue
  Given a bot in SAFE_MODE or circuit-breaker-exhausted state
  When the Close Worker triggers bot_safe_close
  Then the HL short is fully closed first (size = 0 confirmed)
  And vault.executeStrategy(RedeemStrategyV1) closes the LP position
  And RedemptionQueue.createRequest is called, enqueuing the HL portion payout
  And Operator calls fulfillRequest(tokens, amounts) — FIFO pop, funds transferred on-chain to user
  And bots.status transitions to 'closed'
  And investor receives notification "Bot safely closed. Funds have been returned to your wallet."

AC-EXBOT-009-2: Edge case — hedge close fails, LP not touched
  Given a bot_safe_close is triggered
  When the HL short close fails after 3 retries
  Then close_operations.state holds at 'hedge_close_pending'
  And LP position is NOT closed until hedge is confirmed at size=0
  And admin escalation notification is sent
  And bots.status transitions to 'safe_mode'

AC-EXBOT-009-3: Edge case — 3rd bot_safe_close trigger within 7 days
  Given a bot has already had 2 bot_safe_close events within the last 7 days
  When a 3rd bot_safe_close is triggered
  Then the close flow executes normally (hedge → LP → RedemptionQueue)
  And admin escalation notification is sent concurrently
  And bots.status transitions to 'closed' after fulfillRequest completes

AC-EXBOT-009-4: Negative path — fulfillRequest not yet called by Operator
  Given close_operations.state = 'redemption_queued'
  When the Operator has not yet called fulfillRequest
  Then the redemption request remains enqueued (not lost)
  And bots.status remains 'closing' until fulfillRequest completes
  And user funds are NOT at risk — request is on-chain and cannot be dropped
```

## Trace

## Notes

bot_safe_close uses hedge-first order: HL short closed → LP closed via executeStrategy(RedeemStrategyV1) → RedemptionQueue.createRequest enqueues HL portion → Operator fulfillRequest pays user FIFO on-chain. No park/redeploy loop. close_operations idempotency key (UNIQUE constraint) prevents double settlement. Multiple pending RedemptionQueue requests per (user, botId) are allowed — FIFO order is enforced by on-chain queue.
